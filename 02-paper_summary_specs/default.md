# 默认总结规范

这份文件是当前默认使用的论文总结规范。

当我收到“按默认规范总结并打包 ...”时，默认按本文件执行。
当用户明确说“按规范 `<name>` 总结并打包 ...”时，优先按 `02-paper_summary_specs/<name>.md` 执行。

---

## 1. 输入与输出

### 生成前补充要求

- 无论是单篇论文总结，还是多篇论文对比总结，开始生成正文前都应先弹出一个对话框，让用户输入本次特定的总结要求、关注点和关键问题。
- 用户可以留空；留空时完全按默认规范执行。
- 用户输入的要求应与默认规范合并使用：本次要求优先决定关注重点，默认规范保证结构、图片、公式、打包和回写要求不丢失。
- 用户输入不能被当作论文事实来源；如果原文没有直接回答某个问题，必须明确说明“原文未直接说明”。

### 输入

- 输入对象默认是 `wzy/01-paper/` 下的一个或多个 PDF。
- 用户可以只给 PDF 路径，不必手动指定输出文件名。

### 输出

- 生成一份中文 Markdown 总结。
- 完成后按 `paper workflow` 打包。
- 最终交付物是打包目录中的同名 Markdown 文件。

---

## 2. 自动命名规则

### 单篇论文

- 总结 Markdown 文件名按当前样例命名风格自动生成：
  - `第一作者-年份-英文标题-期刊或来源信息.md`
- 例如：
  - `Warttmann-2026-Suppressing-crosstalk-for-Rydberg-quantum-gates-Quantum.md`
  - `Cheng-2025-Density-dependent-gauge-field-Raman-lattices-PhysRevA-111-013319.md`

### 多篇论文

- 如果一次指定多个 PDF，默认生成一份合并总结。
- 文件名按同样风格自动生成一个简洁的合并名称。
- 用户不需要在指令里再手动输入输出文件名，除非想显式覆盖。

### 打包目录

- 打包目录名称必须与 Markdown 文件名去掉 `.md` 后完全一致。

---

## 3. 默认文档结构

注：图一定要在对应章节出现，一定不能放到后面统一列举！！！

默认参考 `Cain et al. 2026 - Shor's algorithm is possible with as few as 10,000 reconfigurable atomic qubits` 的结构组织，优先采用以下顺序：

1. 标题
2. 论文信息(包含md文档更新日期)
3. 一句话结论
4. 摘要翻译（如果原文摘要是中文，就直接放原文）
5. 这篇文章在解决什么问题 / 主要在解决什么问题
6. （这部分是主要内容）按章节总结，内容上：
   1. 要求先总结这个章节的主要内容和说明章节作用作用，再对章节中每一段的内容都要单独提到并做出总结。
   2. 章节中对应出现的图和公式也要列出来并详细解释，如果一张图不同子图对应不同章节的内容，可以多次引用图片。重点：这部分涉及到图片的部分需要及时放在文档对应章节的地方。公式在其出现的章节也要对应详细解释。
   3. 对于核心或者重要内容，要额外给出详细的推导和解释。
   4. 如果原文有 `Method`、`Methods`、`Materials and Methods`、附录中的方法学部分、补充材料中的技术细节等，按章节总结时必须纳入并详细分析，不能只概括主文结论而跳过方法部分。
7. 文章的核心贡献
8. 这篇文章最值得记住的物理点
9. 总结这篇文章的最主要声称点，并且详细讲清楚其通过什么方式，观察到什么结果，以及结果如何支撑他声称的论点的。
10. 可能的局限与延伸
11. 个人整理版结论
    1. 全文内容的总结
    2. 附上一段锐评：客观评价文章相比于之前成果的创新性，对于整个领域究竟有没有作用。
    3. 思考一下文章任何可能的后续拓展，2点以上，越多越好。
12. 一些文中的关键名词解析（不少于3个）。详细解释其原理以及在文中的作用，最好使用简单的例子，尽量配图进行说明。
13. 列出文中引用的关键文献（不少于5篇），介绍文章主要做了什么以及和本文的关系，并尽可能挑选一张原文的代表图来说明。并附上文章引用信息和链接。
14. 关键问题问答。这一部分放在最后，先根据AI自己的理解以问答的形式添加2个关键问题和解答。供后续精读文章时根据读者的理解再添加。有必要可以附图。
15. 完成总结后，根据总结的内容为文章增加五个关键词作为tags，添加在论文信息中的“关键tags”这一项中

说明：

- 如果原文结构特殊，可以调整标题表述，但总体结构尽量保持一致。
- 如果是多篇论文合并总结，优先使用 `02-paper_summary_specs/compare.md` 的多篇对比规范；如果仍按本规范合并总结，应在末尾增加“对比/联系/差异”部分。
- “摘要翻译”默认紧跟在“一句话结论”之后：
  - 如果原文摘要是英文或其他外文，就翻成中文。
  - 如果原文摘要本身就是中文，就直接放原文。

## 4. 写作要求

- 用中文写。
- 保留必要英文术语。
- 不是机械翻译，要围绕“问题-方法-结果-意义”做总结。
- 按原文章节顺序展开。
- 在“一句话结论”之后，必须补一节“摘要翻译”。
- “按章节总结”默认不仅包括主文正文，也包括与主结论直接相关的 `Method`、`Methods`、`Materials and Methods`、附录和补充材料中的方法学部分；如果这些部分承载关键推导、实验细节、误差模型、资源估算、算法流程或数据处理方法，必须单独展开分析。
- 要区分：
  - 研究背景
  - 模型或方法
  - 关键推导
  - 数值/实验结果
  - 物理意义
  - 局限性
- 最后的“个人整理版结论”需要是归纳总结，但不能脱离原文内容。

---

## 5. 公式要求

- 总结必须尽量包含文中的关键公式。
- 优先使用 Markdown 数学块直接写出公式。
- 行内公式必须使用 `$...$`，不要用反引号包裹公式、变量、态矢、参数、比例关系等数学内容。
- 行间公式必须写成稳定可渲染的块级数学格式。
- 多行公式必须放在 `$$ ... $$` 中，并优先使用 `aligned`、`cases`、`pmatrix` 等环境组织换行。
- 不要写 `+=`、裸换行后直接续写整条公式、或其他容易让 Markdown 预览错误显示的中间态写法。
- 如果公式较复杂或 PDF-only 场景下不便可靠转写，可以补充自动提取的公式图片。
- 至少应覆盖：
  - 模型 Hamiltonian
  - 低能有效模型/映射关系
  - 关键误差公式、保真度公式、相变判据、能谱条件等
- 默认允许把公式集中放在“文中公式”区块中。
- 如果用户明确要求，也可以把公式分散放回对应章节。

---

## 6. 图片要求

- 总结必须尽量包含文中的主要图片。
- 每个 `Fig.` 小节下应直接嵌入对应图片，而不是只写文字说明。
- 每张图至少要解释：
  - 图中画了什么
  - 这张图想说明什么
  - 它和全文主结论的关系
- 如果 PDF 自动提取把图拆成多个 panel，可以保留多个 panel。
- 如果自动提取不稳定，可以退回到：
  - 原始 figure 文件
  - 页面截图
  - PDF 自动提取图片的组合
- 最后把所有提取出来的文中的图作为附录放在最后，并附上对应的figure caption
  - 做完这一步之后检查一下章节中的图片引用，不要出现引用整页PDF文档作为图片的情况。


---

## 7. 打包要求

完成总结后，必须按 `paper workflow` 打包。

打包目录中必须包含：

- 同名 Markdown
- `sources/`
- `extracted/`
- `images/`
- `manifest.json`

相关材料应尽量完整保存，包括：

- 原始 PDF 文档
- Markdown 中实际引用的图片
- PDF 自动提取出的页面、纯文本、图片和元数据

确保正文中图片都有正确的引用。

---

## 8. 文件命名要求

- 所有相关文件应采用可读、稳定、避免冲突的命名方式。
- 图片文件名应尽量包含用途信息，例如：
  - `fig-01-...`
  - `formula-01-...`
  - `pdf01-fig-...`
- 多 PDF 情况下，自动提取材料前缀要带 `pdf01`、`pdf02` 等编号。

---

## 9. 用户历史要求汇总

这部分汇总你之前明确提出过的要求，后续可以继续在这里增删：

- 指定一个或多个 PDF 时，输出一份包含所有图片和公式的总结 Markdown。
- 总结完成后，把所有相关材料一起打包到 `wzy` 下的一个子文件夹中。
- 子文件夹名称必须和 Markdown 文件名一致。
- 所有相关文件都要按合适的方式命名。
- 默认规范文件和自定义规范文件都放在 `wzy` 下的固定子文件夹中。
- 尽量不要让我在指令里手动输入规范文件全路径和输出文件名。
- 总之，用户输入指令应尽可能短。

---

## 10. 最短指令约定

今后默认使用下面这两类短指令：

### 用默认规范

```text
按默认规范总结并打包 01-paper/xxx.pdf
```

### 用指定规范

```text
按规范 <name> 总结并打包 01-paper/xxx.pdf
```

其中：

- `<name>` 默认解析为 `02-paper_summary_specs/<name>.md`
- 用户不需要写完整路径
- 用户不需要手动指定输出文件名，默认自动生成

多 PDF 也同理：

```text
按默认规范总结并打包 01-paper/a.pdf 和 01-paper/b.pdf
```

---

## 11. 覆盖规则

如果用户在任务里显式提出新的结构或内容要求，则以该次任务要求为准。
如果用户之后修改了本文件，则新的文件内容自动成为新的默认规范。
每次开始生成总结前，都必须重新读取磁盘上的本文件最新内容，不能沿用之前会话里记住的旧版本。
